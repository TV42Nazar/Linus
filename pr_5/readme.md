#Завдання 2.15

Реалізувати циклічну структуру даних, де звільнення одного вузла робить інші недосяжними для cleanup.

---

### 1. Встановлення необхідних інструментів

Перш за все, потрібно оновити список пакетів і встановити Valgrind

```bash
sudo apt update
sudo apt install valgrind

```

---

### 2. Команди компіляції та запуску

```bash
# 1. Компіляція програми
gcc -g cyclic_leak.c -o cyclic_leak

# 2. Перевірка на витоки пам'яті
valgrind --leak-check=full --show-leak-kinds=all ./cyclic_leak

```
---

### 3. Пояснення команди компіляції N2

#### --leak-check=full:
- За замовчуванням Valgrind видає лише резюме. Цей прапорець змушує його робити повний звіт про кожен окремий витік пам'яті, який він знайде.

#### --show-leak-kinds=all:
#### Valgrind ділить витоки на типи:

- Definitely lost
  Пам'ять точно втрачена (наш випадок із вузлом B).

- Indirectly lost
  Пам'ять втрачена через те, що вказівник на неї був у іншому втраченому блоці.

- Possibly lost
  Можливий витік (вказівник перемістився всередину блока).

- Still reachable
  Пам'ять не звільнена, але вказівники на неї ще є.

#### --track-origins=yes:

- Якщо програма використовує неініціалізовану змінну, цей параметр покаже, де саме в коді цю змінну було створено (звідки взялося "сміття").

## 4. Опис рішення

Реалізовано приклад **циклічного витоку пам'яті**, що виникає через порушення логіки очищення зв'язних структур даних.

### Алгоритм відтворення витоку:

1. У динамічній пам'яті (купі) створюються два об'єкти типу `Node`.
2. Об'єкти налаштовуються так, що вони посилаються один на одного, утворюючи замкнене кільце.
3.  * Виконується звільнення лише одного вузла (`nodeA`).
* Вказівник на другий вузол (`nodeB`) зануляється в локальній області видимості.

### Наслідки:

* **Dangling Pointer:** Вузол `nodeB` продовжує зберігати вказівник на `nodeA`, який вже є невалідним (вивільненим).
* **Недосяжність пам'яті:** Оскільки єдиний шлях до `nodeB` лежав через видалений `nodeA`, а локальний вказівник було занулено, пам'ять, яку займає `nodeB`, стає повністю недосяжною для програми.

### Аналіз інструментарієм:

Інструмент діагностики зафіксує витік класу **"definitely lost"**. Це пояснюється тим, що в системі не залишилося жодного вказівника, який би містив адресу цього блоку пам'яті, що унеможливлює його подальше звільнення.
