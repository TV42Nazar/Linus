# Завдання 4.1

Скільки пам’яті може виділити `malloc(3)` за один виклик?
Параметр `malloc(3)` є цілим числом типу даних `size_t`, тому логічно максимальне число, яке можна передати як параметр `malloc(3)`, — це максимальне значення `size_t` на платформі `(sizeof(size_t))`. У 64-бітній Linux size_t становить 8 байтів, тобто 8 * 8 = 64 біти. Відповідно, максимальний обсяг пам’яті, який може бути виділений за один виклик `malloc(3)`, дорівнює 2^64. Спробуйте запустити код на `x86_64` та `x86`. Чому теоретично максимальний обсяг складає 8 ексабайт, а не 16?

---


## Опис роботи
- На `x86_64` ліміт виділення обмежений 8 ЕБ через поділ простору User/Kernel та обмеження типів вказівників.

Максимальне число для malloc — це `SIZE_MAX`. На 64-бітній системі це $2^{64}-1$ (16 ЕБ). Проте в Linux віртуальна пам'ять розділена: старший біт (найвищий) зарезервований під простір ядра (Kernel space). Таким чином, користувачеві доступна лише половина адресного простору — $2^{63}$ байтів (8 ЕБ).

```bash
void *ptr = NULL;
while (<some-condition-is-true>) {

        ptr = malloc(n);
    [... <використання 'ptr'> ...]
    free(ptr);
}
```

## Опис роботи

У наданому коді після `free(ptr)` вказівник `ptr` не занулюється. Це призводить до помилки на наступній ітерації циклу, оскільки умова `if (!ptr` не спрацьовує (вказівник не NULL, хоча пам'ять звільнена). Програма завершиться аварійно.

---

## Опис роботи

Я використав механізм `linker wrapping`, щоб перехопити виклики `malloc` та `free`. Всі виділені адреси я записую у масив структур, а при звільненні — помічаю їх як очищені. У фіналі просто фільтрую масив: ті блоки, що не мають відмітки про звільнення, вважаю витоками пам'яті
