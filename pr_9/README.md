**Завдання 9.1**
**Опис завдання:** Напишіть програму, яка читає файл `/etc/passwd` за допомогою команди `getent passwd`, щоб дізнатись, які облікові записи визначені на вашому комп’ютері. Програма повинна визначити, чи є серед них звичайні користувачі (ідентифікатори UID повинні бути більші за 1000), окрім вас.

**Команди запуску:**
```bash
chmod +x task9_1.sh
./task9_1.sh
```

**Код:**
```bash
#!/bin/bash

# Отримуємо ім'я поточного користувача
CURRENT_USER=$(whoami)

echo "Список звичайних користувачів (UID >= 1000), окрім $CURRENT_USER:"
echo "--------------------------------------------------------"

# Використовуємо getent і awk для фільтрації
# $1 - ім'я користувача, $3 - UID
getent passwd | awk -F: -v curr="$CURRENT_USER" '
    $3 >= 1000 && $3 < 60000 && $1 != curr {
        printf "Користувач: %-15s UID: %s\n", $1, $3
    }
'
```

**Короткий опис рішення:**
Скрипт отримує ім'я користувача, який його запустив. Далі команда `getent passwd` передає вивід утиліті `awk`, яка розбиває рядки по символу двокрапки (`:`). `awk` перевіряє, щоб UID (третє поле) був більшим або рівним 1000 (стандарт для звичайних користувачів у більшості сучасних дистрибутивів), меншим за 60000 (щоб відсіяти системні/nobody акаунти), і щоб ім'я не збігалося з поточним користувачем.

**Завдання 9.3**
**Опис завдання:** Програма від імені root копіює файл, створений звичайним користувачем, у домашній каталог. Далі звичайний користувач намагається його змінити та видалити.

**Команди запуску:**
```bash
chmod +x task9_3.sh
./task9_3.sh
```

**Код:**
```bash
#!/bin/bash

FILE_NAME="test_file_9_3.txt"
HOME_FILE="$HOME/$FILE_NAME"

# 1. Створення від імені звичайного користувача
touch /tmp/$FILE_NAME
echo "Створено тимчасовий файл: /tmp/$FILE_NAME"

# 2. Копіювання від імені root у домашній каталог
sudo cp /tmp/$FILE_NAME $HOME_FILE
echo "Файл скопійовано від імені root: $HOME_FILE"
ls -l $HOME_FILE

# 3. Спроба зміни звичайним користувачем
echo -e "\nСпроба запису у файл:"
echo "Новий текст" >> $HOME_FILE 2>/dev/null && echo "-> Запис успішний!" || echo "-> Помилка: відмовлено в доступі (Permission denied)."

# 4. Спроба видалення звичайним користувачем
echo -e "\nСпроба видалення файлу:"
rm -f $HOME_FILE 2>/dev/null && echo "-> Файл успішно видалено!" || echo "-> Помилка видалення."
```

**Короткий опис рішення:**
Під час спроби запису у файл користувач отримає `Permission denied`, оскільки власником скопійованого файлу став `root` (через використання `sudo cp`), і права на запис є лише у власника.
Під час спроби видалення файл **буде успішно видалено**. Це відбувається тому, що в Linux право на видалення файлу визначається правами доступу до *каталогу*, у якому він знаходиться (у даному випадку — домашній каталог користувача), а не правами на сам файл.

**Завдання 9.5**
**Опис завдання:** Створення тимчасового файлу, зміна власника/прав від суперкористувача (`chown`, `chmod`) та визначення можливостей читання/запису.

**Команди запуску:**
```bash
chmod +x task9_5.sh
./task9_5.sh
```

**Код:**
```bash
#!/bin/bash

TEST_FILE="perms_test.txt"
touch $TEST_FILE

echo "Початкові права (власник $USER):"
ls -l $TEST_FILE

# Змінюємо власника на root і права на читання/запис тільки для root (600)
sudo chown root:root $TEST_FILE
sudo chmod 600 $TEST_FILE

echo -e "\nПрава змінено на 600, власник root:"
ls -l $TEST_FILE
[ -r $TEST_FILE ] && echo "-> Читання: ДОЗВОЛЕНО" || echo "-> Читання: ЗАБОРОНЕНО"
[ -w $TEST_FILE ] && echo "-> Запис: ДОЗВОЛЕНО" || echo "-> Запис: ЗАБОРОНЕНО"

# Змінюємо права на 644 (root: rw-, всі інші: r--)
sudo chmod 644 $TEST_FILE

echo -e "\nПрава змінено на 644, власник root:"
ls -l $TEST_FILE
[ -r $TEST_FILE ] && echo "-> Читання: ДОЗВОЛЕНО" || echo "-> Читання: ЗАБОРОНЕНО"
[ -w $TEST_FILE ] && echo "-> Запис: ДОЗВОЛЕНО" || echo "-> Запис: ЗАБОРОНЕНО"

# Прибираємо за собою
rm -f $TEST_FILE
```

**Короткий опис рішення:**
Скрипт створює файл, передає його у власність root і встановлює права `600` (`-rw-------`). Оскільки поточний користувач не є `root`, конструкції `[ -r ... ]` та `[ -w ... ]` (перевірки Bash на доступність читання і запису) повертають хибу. Після зміни прав на `644` (`-rw-r--r--`), файл стає доступним для читання "іншим" користувачам, тому перевірка `[ -r ]` стає істинною, але запис усе ще заборонений.

**Опис завдання:** Переглянути права на `~`, `/usr/bin` та `/etc`. Здійснити спроби обійти ці права (читання, запис, виконання).

**Команди запуску:**
```bash
chmod +x task9_6.sh
./task9_6.sh
```

**Код:**
```bash
#!/bin/bash

echo "=== Аналіз прав доступу ==="
ls -ld ~ /usr/bin /etc

echo -e "\n=== Спроба ЗАПИСУ ==="
echo "1. Запис у домашній каталог (~/test_file):"
touch ~/test_file && echo "-> УСПІХ: файл створено." && rm ~/test_file || echo "-> ПОМИЛКА."

echo "2. Запис у каталог /etc (/etc/test_file):"
touch /etc/test_file 2>/dev/null && echo "-> УСПІХ." || echo "-> ПОМИЛКА: відмовлено в доступі (немає прав на запис у директорію)."

echo -e "\n=== Спроба ЧИТАННЯ ==="
echo "1. Читання публічного файлу (/etc/passwd):"
head -n 1 /etc/passwd > /dev/null 2>&1 && echo "-> УСПІХ: файл прочитано." || echo "-> ПОМИЛКА."

echo "2. Читання захищеного файлу (/etc/shadow):"
head -n 1 /etc/shadow 2>/dev/null && echo "-> УСПІХ." || echo "-> ПОМИЛКА: відмовлено в доступі."

echo -e "\n=== Спроба ВИКОНАННЯ ==="
echo "1. Виконання бінарного файлу (/usr/bin/whoami):"
/usr/bin/whoami > /dev/null 2>&1 && echo "-> УСПІХ: програма виконалась." || echo "-> ПОМИЛКА."
```

**Короткий опис рішення:**
Каталоги `/usr/bin` та `/etc` належать root і не мають прав на запис для звичайних користувачів (зазвичай `755`), тому створення файлів там блокується на рівні ядра ОС. Читати можна лише ті файли в `/etc`, які мають прапорець `r` для всіх користувачів (як `/etc/passwd`), тоді як `/etc/shadow` має права `640` (root/shadow) або `600`, що унеможливлює його читання. Виконання файлів у `/usr/bin` дозволено завдяки прапорцю `x` для всіх (`rwxr-xr-x`).

**Завдання 15**
**Опис завдання:** Напишіть скрипт, що веде логування команд користувача. Як приховати його від `ps`?

**Команди запуску:**
```bash
# Для запуску як прихованого процесу (читайте опис)
exec -a systemd-worker ./logger.sh
```

**Код (`logger.sh`):**
```bash
#!/bin/bash
# Простий проксі-скрипт (Custom Shell), який записує все, що вводиться

LOG_FILE="/tmp/user_hidden_log.txt"

echo "Оболонка запущена. Введіть 'exit' для виходу."

# Використовуємо вбудований цикл read для імітації терміналу
while read -re -p "$USER@$HOSTNAME:~$ " cmd; do
    if [ "$cmd" == "exit" ]; then
        break
    fi
    
    # Логуємо команду перед її виконанням
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] Команда: $cmd" >> "$LOG_FILE"
    
    # Виконуємо команду
    eval "$cmd"
done
```

**Короткий опис рішення та приховування від `ps`:**
Цей скрипт імітує поведінку командної оболонки: він приймає команди від користувача, записує їх у файл журналу `/tmp/user_hidden_log.txt` і виконує через `eval`.
**Як приховати від `ps`:**
1. **Підміна назви процесу в аргументі `argv[0]`:** При запуску скрипта використовується утиліта `exec` з прапорцем `-a`. Команда `exec -a systemd-worker ./logger.sh` замінює поточний процес оболонки на скрипт, але інформує ядро, що назва процесу — `systemd-worker`. Якщо системний адміністратор виконає `ps aux | grep logger`, він його не знайде. Процес маскуватиметься під легітимний системний демон.
2. **Альтернативні (більш складні) варіанти:** Використання rootkit на рівні ядра Linux (через модулі LKM), які перехоплюють системний виклик `getdents64`, приховуючи каталог PID процесу у віртуальній файловій системі `/proc`. Таким чином процес взагалі зникає не лише з `ps`, а й з `top` чи `htop`.
