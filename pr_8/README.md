## Виконані завдання

### Завдання 8.1
**Чи може виклик `write()` повернути значення, відмінне від `nbytes`?**
Так, може. Це називається «частковим записом» (partial write). Така ситуація є штатною і може виникнути з кількох причин:
1. **Переривання сигналом** - процес був перерваний обробником сигналу до того, як усі байти були записані.
2. **Нестача місця на диску** або досягнення дискової квоти користувача.
3. **Неблокуючий ввід/вивід (O_NONBLOCK)** - якщо ми пишемо у канал (pipe), сокет або термінал, і внутрішній буфер ядра заповнюється швидше, ніж приймаюча сторона його читає, `write()` запише лише те, що влізе, і поверне кількість успішно записаних байтів.


**Команди для компіляції та запуску**
```bash
gcc -Wall task8_1.c -o task8_1
./task8_1
```

### Завдання 8.2
**Читання після зсуву файлового покажчика.**
Файл містить байти (індекси від 0 до 9):
`[0]:4, [1]:5, [2]:2, [3]:2, [4]:3, [5]:3, [6]:7, [7]:9, [8]:1, [9]:5`

Виклик `lseek(fd, 3, SEEK_SET)` переміщує покажчик на зміщення 3 від початку файлу (тобто на четвертий байт, який дорівнює `2`).
Наступний виклик `read(fd, &buffer, 4)` зчитує 4 байти, починаючи з цієї позиції.
**Результат:** буфер міститиме байти: `2, 3, 3, 7`.


**Команди для компіляції та запуску**
```bash
gcc -Wall task8_2.c -o task8_2
./task8_2
```

### Завдання 8.3
**Дослідження `qsort` та перевірка правильності.**
Бібліотечна функція `qsort` в сучасних системах Linux (glibc) зазвичай під капотом реалізує MergeSort (якщо є пам'ять) або оптимізований QuickSort. Щоб знайти "найгірший" варіант, програма рахує кількість викликів функції-компаратора. Найбільше порівнянь зазвичай відбувається на масивах, де всі елементи однакові, або на зворотно відсортованих масивах.


**Команди для компіляції та запуску**

### Завдання 8.4
**Аналіз виклику `fork()`.**
Виклик `fork()` створює точну копію поточного (батьківського) процесу. Після цього обидва процеси продовжують виконання з наступного рядка.
Функція `fork()` повертає:
- `0` для новоствореного дочірнього процесу.
- `PID` (ідентифікатор, число більше нуля) дочірнього процесу для батьківського процесу.

Тому програма виведе два рядки (порядок може змінюватися залежно від планувальника ОС ядра): один міститиме `0`, а інший — ідентифікатор дочірнього процесу (наприклад, `34512`).

**Команди для компіляції та запуску**
```bash
gcc -Wall task8_4.c -o task8_4
./task8_4
```

### Завдання 15
**Атомарний обмін вмістом двох файлів без створення тимчасових.**
Звичайний спосіб заміни файлів вимагає створення третього тимчасового файлу (`mv A temp; mv B A; mv temp B`), але це не є атомарною операцією (система може "впасти" посеред процесу).
В Linux існує сучасний системний виклик `renameat2` з прапорцем `RENAME_EXCHANGE`, який дозволяє ядру миттєво обміняти імена двох файлів на рівні файлової системи (перезаписуючи їхні `inode` посилання).


**Команди для компіляції та запуску**
```bash
# Підготовка тестових файлів
echo "Це вміст ПЕРШОГО файлу" > a.txt
echo "Це вміст ДРУГОГО файлу" > b.txt

# Компіляція
gcc -Wall task15.c -o task15

# Запуск обміну
./task15 a.txt b.txt

# Перевірка (тепер a.txt містить текст другого файлу)
cat a.txt
cat b.txt
```

### Висновок
Під час виконання цієї частини роботи було детально досліджено системні виклики `write`, `read`, та `lseek`. Доведено, що системні виклики можуть повертати неочікувані (але задокументовані) значення, як у випадку з частковим записом у неблокуючий буфер. Також було досліджено створення нових процесів за допомогою `fork()`, проаналізовано складність алгоритму `qsort` на різних вибірках даних та успішно застосовано специфічний системний виклик Linux `renameat2` для атомарної роботи з файловою системою.
